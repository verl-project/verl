# Megatron output_processor end-to-end validation in verl

Last updated: 07/11/2026.

This note records the validation that was run for
[verl-project/verl#6933](https://github.com/verl-project/verl/pull/6933) in
response to [NVIDIA/Megatron-LM#4590](https://github.com/NVIDIA/Megatron-LM/issues/4590).
It is intended to be reproducible guidance for developers who want to check that
verl's fused Megatron log-probability and entropy path can use Megatron's native
`output_processor` hook instead of monkey-patching `GPTModel.forward`.

The validation has two separate parts:

- deterministic same-weight, same-batch offline A/B tests, which prove numerical
  equivalence between the legacy monkey-patch path and the native hook path;
- an independent live five-step RL smoke, which proves the hook path survives the
  real verl loop: rollout, actor/rollout weight synchronization, fused
  log-probability and entropy, optimizer update, offload, and checkpoint save.

Do not interpret the live smoke as a bitwise comparison between two training
runs. The two live runs used independent vLLM rollout processes and therefore
sampled different trajectories.

## What Was Covered

The historical live smoke covered this configuration:

| Item | Historical value |
| --- | --- |
| Model | Dense `Qwen/Qwen2.5-0.5B-Instruct` |
| Dataset | GSM8K parquet generated by `examples/data_preprocess/gsm8k.py` |
| Algorithm | GRPO |
| Trainer entry point | `python -m verl.trainer.main_ppo` |
| Model engine | `model_engine=megatron`, Megatron engine v1 |
| Rollout backend | vLLM |
| GPUs | 4x A100-SXM4-80GB |
| dtype | bf16 |
| Actor tensor parallelism | TP=2 |
| Actor pipeline parallelism | PP=1 |
| Rollout tensor parallelism | TP=2 |
| Fused path | `actor_rollout_ref.model.use_fused_kernels=True` |
| Offload | actor param/grad/optimizer offload, ref param offload |
| Training length | 5 steps |
| Checkpoint frequency | `trainer.save_freq=4`; saves were recorded at steps 4 and 5 |

The validation also covered rollout-to-actor probability correlation, dynamic
batching for actor/ref/rollout log-probability, sequence packing/THD shape in the
Megatron engine path, tied output embeddings in the live model, optimizer update,
and distributed checkpoint save.

These cases were not covered:

- BF16 only; FP8 padding was not exercised.
- PP=1; this run does not validate a real pipeline-parallel 1F1B schedule.
- `build_schedule_plan()` / postprocess-node output conversion is outside
  verl#6933.
- MTP was not validated.
- This is a five-step integration smoke, not a convergence or performance study.
- Independent vLLM rollouts are not deterministic across separate runs.
- Checkpoint save was validated. Checkpoint reload/resume was not independently
  verified by this smoke.

`VERL_FUSED_USE_OP_HOOK` is parsed once at model-build time in the current PR:

- unset, empty, or `auto`: use the hook when the bound Megatron forward supports
  `output_processor` and `output_processor_context`; otherwise use the legacy
  path;
- `hook`, `1`, `true`, or `yes`: require the native hook and fail fast if the
  contract is unavailable;
- `legacy`, `0`, `false`, or `no`: force the legacy monkey-patched
  `GPTModel.forward` path.

The historical live run used `VERL_FUSED_USE_OP_HOOK=0` and `=1`. On the current
branch, use `legacy` and `hook` for readability; they are equivalent to those
historical numeric values.

## Compatibility Prerequisite

Megatron-Core 0.18 removed `ModelType.encoder_and_decoder`. The historical
environment used `mbridge==0.15.1`, which still referenced that enum member and
failed before Megatron model construction. That failure happens upstream of any
forward call, so it is independent of the `output_processor` migration.

Track the compatibility issue at
[ISEEKYAN/mbridge#155](https://github.com/ISEEKYAN/mbridge/issues/155) and the
verl dependency note at
[verl-project/verl#6936](https://github.com/verl-project/verl/issues/6936).
When reproducing the historical Megatron-Core 0.18 setup, use a mbridge version
or patch that guards the removed enum. The exact historical local mbridge patch
was not archived in this repository, so the original environment cannot be
reconstructed byte-for-byte from this guide alone. Use a mbridge revision that
has fixed this compatibility issue, or apply an equivalent fix based on the
public issue. Do not treat a `ModelType.encoder_and_decoder` crash as evidence
about the hook path.

```bash

export PYTHON="${VENV_ROOT}/bin/python"

export PYTHONPATH="${VERL_ROOT}:${MEGATRON_ROOT}:${PYTHONPATH:-}"
export CUDA_DEVICE_MAX_CONNECTIONS=1

EOF
```

## Hook Run

Forced hook mode requires Megatron's native forward contract. If the bound model
forward does not accept `output_processor` and `output_processor_context`, the
current PR fails during model setup instead of silently falling back.

```bash
run_live_smoke() {
  local mode="$1"
  local run_name="$2"

  VERL_FUSED_USE_OP_HOOK="${mode}" \
    "${PYTHON}" -m verl.trainer.main_ppo \
    "${COMMON_OVERRIDES[@]}" \
    trainer.experiment_name="${run_name}" \
    trainer.default_local_dir="${OUTPUT_ROOT}/${run_name}"
}

run_live_smoke hook hook_e2e
```

Optional one-off diagnostic: during model setup, check the actual GPT model for
`forward_backup`. In the historical diagnostic run, hook mode reported
`has_forward_backup=False` on all ranks, proving that `patch_fused_forward` left
the native forward untouched and the run used `output_processor`.

This diagnostic should not be a permanent production log. The current forced
`hook` mode already fail-fast checks the forward contract.

## Legacy Control Run

Run the same configuration with a separate output directory:

```bash
run_live_smoke legacy legacy_e2e
```

In the historical diagnostic control, legacy mode reported
`has_forward_backup=True`, which is expected because the legacy path monkey
patches `GPTModel.forward`.

## Deterministic Same-Batch A/B Validation

The offline A/B checks are the numerical-equivalence proof. They used the same
weights and the same batch in two independent processes, one with the legacy path
and one with the hook path. The historical harness files were external validation
artifacts, not production verl code:

```text
ab_test_fused_hook.py
ab_test_fused_engine.py
ab_compare.py
run_ab.sh
build_real_batch.py
```

The command shape was:

```bash
cd /path/to/validation-harness
export PYTHONPATH="${VERL_ROOT}:${PYTHONPATH:-}"
export CUDA_HOME=/usr/local/cuda-12.8

# Synthetic fixed-shape matrix: TP in {1,2}, tied in {1,0}.
bash run_ab.sh
"${PYTHON}" ab_compare.py --workdir ab_work

# Real training-shaped replay batch.
"${PYTHON}" build_real_batch.py e2e_work/real_batch.pt
mkdir -p replay_work
cp e2e_work/real_batch.pt replay_work/batch_tp2.pt

CUDA_VISIBLE_DEVICES=0,1 VERL_FUSED_USE_OP_HOOK=legacy \
  "${VENV_ROOT}/bin/torchrun" --nproc_per_node=2 \
  ab_test_fused_hook.py \
  --tp 2 --tie 1 --tag old --workdir replay_work --build_path real_get_model

CUDA_VISIBLE_DEVICES=0,1 VERL_FUSED_USE_OP_HOOK=hook \
  "${VENV_ROOT}/bin/torchrun" --nproc_per_node=2 \
  ab_test_fused_hook.py \
  --tp 2 --tie 1 --tag new --workdir replay_work --build_path real_get_model

"${PYTHON}" ab_compare.py --workdir replay_work
```

The engine-call-shape replay used `ab_test_fused_engine.py` with the same
real-batch source and compared nested/THD outputs by values.

Historical pass criteria and recorded results:

| Offline case | Coverage | Recorded result |
| --- | --- | --- |
| Synthetic matrix | TP=1/2, tied and untied output embeddings, output-weight branches, sequence-parallel gather | `max_abs(log_probs)=0`, `max_abs(entropy)=0` |
| Real `get_model` build path | Same weights and batch through the real Megatron model construction path | bit-identical |
| After vLLM install | Regression against the same TP=2 tied case | bit-identical |
| Real training-shaped replay | GSM8K vLLM rollout batch, variable padding, THD `padding_causal`, GRPO advantages | `max_abs(log_probs)=0`, `max_abs(entropy)=0`, `max_abs(pg_loss)=0` |
| Engine path replay | Megatron engine nested/THD call shape, TP=1/2, tied embeddings | `max_abs(log_probs)=0`, `max_abs(entropy)=0`, memory delta 0 |

The current PR also includes focused tests:

```bash
pytest -s -x tests/models/test_model_forward_fused.py
```

Those tests cover mode selection, forced-hook failure, fixed model-build-time
mode, `temperature` through `output_processor_context`, sequence-parallel gather,
and tied/untied output-weight resolution. They are focused unit tests, not a
replacement for the offline A/B or live smoke.

## Expected Live-Run Checks

A passing live smoke should show:

- both hook and legacy runs reach `Training Progress: 100%` for `5/5`;
- both runs log actor update timing, for example `timing_s/update_actor`;
- both runs log rollout-to-actor weight update timing, for example
  `timing_s/update_weights`;
- both runs save checkpoints under distinct output directories, including
  `global_step_4` and the final `global_step_5`;
- no NaN/Inf in the metric scan;
- rollout/actor probability correlation remains high. The historical logs
  recorded values around `0.9996` on each step;
- hook mode has no `forward_backup` in the optional diagnostic;
- legacy mode has `forward_backup` in the optional diagnostic.

Historical live-smoke metrics were sane but not bit-identical across modes:

| Metric | Legacy run | Hook run |
| --- | --- | --- |
| `actor/entropy`, 5 steps | 0.542, 0.524, 0.617, 0.485, 0.489 | 0.595, 0.657, 0.471, 0.432, 0.597 |
| `rollout_corr/training_log_ppl`, 5 steps | 0.512, 0.548, 0.570, 0.432, 0.477 | 0.573, 0.623, 0.448, 0.401, 0.544 |
| `actor/pg_loss` | 0 to 0.067 | -0.002 to 0 |
| `actor/grad_norm` | 0 to 0.82 | 0 to 0.57 |
| actor GPU peak | about 8.2 GB | about 8.0 GB |
| CPU memory used with full offload | about 111 GB | about 113 GB |

The historical logs recorded checkpoint save completion at steps 4 and 5. The
local checkpoint directories may have been cleaned up after the run, so use the
training logs as the durable evidence for that historical smoke.

## Interpreting Live-Run Differences

The live hook and legacy runs are two separate RL runs, not two evaluations of
the same sampled batch. vLLM rollout was not reproducible across independent
processes in the historical environment, even with the same nominal seed. That
means reward, entropy, log-ppl, loss, and sequence-length metrics can differ
because the two runs sampled different trajectories.

Use the live smoke to answer "does the full integrated system run through the
hook path?" Use the offline same-weight, same-batch A/B to answer "is the hook
path numerically equivalent to the legacy fused path?"

## Troubleshooting

If `VERL_FUSED_USE_OP_HOOK=hook` fails during model setup with a message about
`output_processor` and `output_processor_context`, the bound Megatron model does
not expose the native hook contract. Use Megatron-Core with the hook support
available from the 0.18 line, or use `VERL_FUSED_USE_OP_HOOK=auto` / `legacy` for
older Megatron versions.

If model construction fails with `ModelType.encoder_and_decoder`, fix the
mbridge / Megatron-Core 0.18 compatibility issue first. This is a dependency
compatibility failure before forward execution, not a hook failure.

If Transformer Engine import fails with a missing CUDA 13 library while using
CUDA 12.8 PyTorch, check that `transformer_engine_cu12` is installed and that the
torch binding was built against the active PyTorch. The historical run required
rebuilding `transformer_engine_torch==2.10.0` with `--no-build-isolation`.

If THD packed attention reports that no backend is available, install a
compatible flash-attn build. The historical run used `flash-attn==2.8.3` built
against PyTorch 2.9.1 and CUDA 12.8.

If hook mode unexpectedly shows `forward_backup=True`, check that
`VERL_FUSED_USE_OP_HOOK` is exported before worker/model initialization and that
the run is not reusing a model object already initialized in legacy mode.

If hook and legacy live metrics differ step by step, do not immediately treat
that as a numerical regression. First check whether the trajectories differ. Use
same-weight, same-batch offline A/B for numerical equivalence.

If checkpoints from hook and legacy overwrite each other, give each mode a
distinct `trainer.default_local_dir` and `trainer.experiment_name`.

If Ray or vLLM state leaks between attempts, stop the old cluster and clear the
old worker processes before rerunning. The historical smoke ran hook and legacy
as independent processes.

If a rebased local smoke fails before model construction with `ModuleNotFoundError:
No module named 'nvtx'`, check profiler overrides. A later local smoke attempt
hit this when the runtime selected an nsys/nvtx profiler path before reaching the
Megatron model; that run did not validate the hook path.
